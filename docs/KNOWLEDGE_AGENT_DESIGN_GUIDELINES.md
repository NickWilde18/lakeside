# Knowledge Agent Design Guidelines

## 目标

规范 Lakeside 中 knowledge agent 的设计方式，避免后续出现：

- knowledge domain 划分混乱
- 路由只靠模糊 description
- 权限控制误写在 prompt 里
- 一个知识问题只能查一次

本文档面向：

- 知识库设计
- knowledge agent 配置
- 顶层 / 领域路由策略

## 一、先分清三个概念

### 1. 知识库

底层存储与检索单元，例如一个 `kb_id` 或一组文档集合。

### 2. knowledge agent

面向上层 agent 平台暴露的“知识专家”。

它不只是“调用一次 retrieve”，还负责：

- 决定如何检索
- 必要时多次检索
- 汇总答案
- 输出引用来源

### 3. 路由策略

顶层 / 领域 Supervisor 决定“当前问题该交给哪个 knowledge agent”。

## 二、知识域怎么划分

生产上不要只按部门名划分。

优先按以下 4 个维度考虑：

### 1. 目标用户

- 普通师生
- 学生助理
- 正式 staff
- 特定内部团队

### 2. 权限模型

- 公开
- 内部
- 强隔离

### 3. 问题类型

- FAQ
- 操作指南
- 内部规范
- 流程制度
- 引导话术

### 4. 回答口径

- 直接给终端用户
- 给助理参考
- 给内部处理人员执行

## 三、什么情况下应该拆开

建议拆开：

- 权限模型不同
- 受众不同
- 文档维护者不同
- 更新节奏不同
- 回答风格明显不同

不建议为测试方便就长期拆开：

- 只是文件格式不同
- 只是上传人不同
- 只是临时想验证路由

## 四、现阶段更稳的做法

如果仍处探索期，优先做：

- 少量 knowledge agent
- 更强的 metadata / audience / tag 管理

而不是一开始就大量按 `kb_id` 物理拆分。

未来更长期的形态可以是：

- 一个共享 IT 知识库
- 多个 knowledge agent 共享同一知识库
- 通过过滤画像区分可检索范围

例如：

- `it_public_knowledge`
- `it_internal_knowledge`

底层不一定是两个完全独立的 `kb_id`。

## 五、knowledge agent description 怎么写

description 不要只写“回答 XX 问题”。

建议固定包含 4 部分：

### 1. 正向范围

它应该回答什么。

### 2. 负向排除

它不应该回答什么。

### 3. 目标用户

面向谁。

### 4. 典型问题

至少给 2 到 5 个代表例子。

## 六、推荐模板

### 公共 IT 知识 agent

```yaml
description: >
  面向普通师生的校园 IT 公开知识库。
  负责回答 VPN、邮箱、账号、校园网、常见信息系统使用说明与基础排障问题。
  不负责学生助理内部规范、群组邮箱地址、内部操作手册，也不直接提交工单。
  典型问题：VPN 怎么连接、校园邮箱怎么配置、WiFi 连上但打不开网页。
```

### 内部学生助理知识 agent

```yaml
description: >
  仅面向 ITSO 学生助理的内部知识库。
  负责回答学生助理手册、内部操作规范、引导话术、学生群组邮箱地址、语料库收集标准等内部问题。
  不负责普通师生公开 FAQ，也不替代正式 ITSM 工单处理。
  典型问题：大学语料库收集标准是什么、某类咨询该如何引导、学生群组邮箱地址在哪里查。
```

## 七、description 不是权限系统

这是最重要的约束之一。

description 只能用于：

- 路由提示
- 范围声明

它不能替代：

- 下游 RAG 的租户隔离
- 上游显式角色判断
- 服务端 ACL

如果某个知识域真的有权限要求，应该通过以下方式保障：

1. 下游 RAG 侧按 `x-user-upn + kb_id` 做隔离
2. Lakeside 路由前先做候选过滤
3. 必要时引入可验证角色，而不是依赖用户自然语言自报身份

## 八、knowledge agent 相比普通 tool 的优势

普通 retrieve tool 适合：

- 单次检索
- 固定输入输出
- 不需要内部自己做决策

knowledge agent 适合：

- 需要多次检索
- 需要 query rewrite
- 需要拆分子问题
- 需要自己组织回答
- 需要稳定输出引用

对 Lakeside 来说，把知识查询做成 agent 的主要收益是：

1. 可内部多次检索
2. 可绑定固定知识域
3. 可统一控制回答风格
4. 更适合顶层 / 领域 Supervisor 编排
5. 后续可继续接 `multiquery`、rerank、answer synthesis

## 九、检索策略建议

knowledge agent 不应被硬编码为“一轮只检索一次”。

推荐能力：

### 1. 查询改写

优先使用官方组件：

- `flow/retriever/multiquery`

### 2. 并列问题拆分

例如：

- “VPN 连不上，学生群组邮箱地址是什么”

不应只查一次，而应按不同意图扩召回。

### 3. 首次命中后补检索

若第一批结果证据不足，允许继续检索更具体问题。

### 4. 引用稳定输出

至少返回：

- `kb_id`
- `doc_id`
- `node_id`
- `filename`
- `snippet`
- `score`

## 十、顶层 / 领域路由建议

不要把所有路由都交给模型自由发挥。

建议采用“硬规则预筛 + 模型选择”：

### 1. 硬规则预筛

先决定哪些 knowledge agent 可以进入候选集。

例如：

- 用户明确说自己是 ITSO 学生助理
- 或会话长期记忆里已有对应角色
- 才允许内部知识 agent 进入候选集

### 2. 模型在候选集中选择

这样比全量开放给模型稳定得多。

## 十一、推荐评测方式

知识路由和知识回答必须有回归评测。

至少维护：

### 1. 路由评测集

每条问题标注：

- 应命中哪个 knowledge agent
- 是否应进入 ITSM

### 2. 检索评测集

每条问题标注：

- 应命中的文档
- 应命中的知识域

### 3. 回答评测集

每条问题检查：

- 是否引用来源
- 是否编造
- 是否答非所问

## 十二、当前适合 Lakeside 的实践建议

在真正进入生产前，建议按下面顺序推进：

1. 先把知识域定义清楚
2. 再写 description 与路由提示词
3. 再验证是否真的需要按 `kb_id` 物理拆分
4. 再补硬规则预筛和权限控制

不要反过来。

## 十三、一句话原则

knowledge agent 的边界，应该由：

- 目标用户
- 权限
- 问题类型
- 回答口径

共同决定。

不要把“测试时临时拆开的知识库”直接当成最终生产架构。
